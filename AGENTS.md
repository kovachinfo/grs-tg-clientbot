# Repository Guidelines

## Project Structure & Current Architecture
The main application lives in [bot_grs.py](/mnt/d/Railway_projects/telegram-gpt-bot2/bot_grs.py). It is a Flask webhook service for Telegram that contains:
- Telegram webhook handling and menu logic
- user management and request limit handling
- OpenAI `Responses API` integration
- conditional `web_search` usage for migration/legal/current-information questions
- PostgreSQL helpers for users, chat history, and news digest storage
- news digest assembly, filtering, rendering, and admin refresh flow

Database bootstrap lives in [init_db.py](/mnt/d/Railway_projects/telegram-gpt-bot2/init_db.py). Connection pooling lives in [database.py](/mnt/d/Railway_projects/telegram-gpt-bot2/database.py) using `ThreadedConnectionPool`.

Deployment entrypoint is [Procfile](/mnt/d/Railway_projects/telegram-gpt-bot2/Procfile):
- `web: gunicorn --bind 0.0.0.0:$PORT bot_grs:app`

Dependencies are in [requirements.txt](/mnt/d/Railway_projects/telegram-gpt-bot2/requirements.txt). Local secrets stay in `.env` and Railway variables.

Useful directories:
- `__pycache__/` autogenerated
- keep experiments and temp scripts out of the root when possible

## Runtime Flow
### Telegram chat flow
1. Telegram sends updates to `POST /webhook/<TELEGRAM_TOKEN>`.
2. The bot resolves the user, language, and request limits from Postgres.
3. For normal chat questions, the bot loads `chat_history`, builds a system prompt, and calls OpenAI via `Responses API`.
4. `web_search` is enabled when the question is current/temporal or migration-related enough to require live data.
5. The bot sends the final answer back through Telegram Bot API and stores the conversation in `chat_history`.

### News digest flow
News digest generation is no longer done per-user on button press.

Current architecture:
- external scheduler: `n8n`
- protected endpoint: `GET/POST /tasks/refresh-news-digest`
- auth: `X-News-Cron-Token` header or `token` query parameter
- language parameter: `lang=ru` or `lang=en`
- optional force parameter: `force=true`

Digest lifecycle:
1. `n8n` calls `/tasks/refresh-news-digest?lang=ru`.
2. The bot asks OpenAI to collect 14-16 candidate articles as JSON.
3. Candidates are normalized locally:
   - full original article URL required
   - relative dates removed
   - generic/blog/category/evergreen pages filtered out
   - duplicates removed
   - max 2 items per domain
4. Candidates are merged into `news_digest_pool`.
5. A final digest is built from the active pool.
6. If quality rules pass, a `ready` snapshot is saved to `news_digests`.

### Admin refresh flow
`/refresh_news` is admin-only and intended for manual testing / forced refresh.
- Admin chat id is controlled by `NEWS_ADMIN_CHAT_ID`
- The command starts a background thread
- While refresh is running, the bot sends `typing`
- At the end it sends a status summary, not the raw digest body

## Database Schema
Current important tables:

### `users`
- `chat_id`
- `language_code`
- `request_count`
- `created_at`
- `is_premium`

Purpose:
- per-user language
- free request limit tracking
- premium override

### `chat_history`
- `chat_id`
- `role`
- `content`
- `created_at`

Purpose:
- short rolling memory for contextual AI answers

### `news_digest_pool`
- `language_code`
- `source_url`
- `source_domain`
- `title`
- `summary`
- `country`
- `article_date_raw`
- `article_date`
- `discovered_at`
- `updated_at`
- `is_active`

Purpose:
- buffered pool of normalized news items
- active source of truth for the current published digest

### `news_digests`
- `language_code`
- `status`
- `items_json`
- `rendered_html`
- `raw_response`
- `model_used`
- `created_at`

Purpose:
- immutable snapshot history of digest generations
- `status` values include `ready`, `draft`, `failed`

### Legacy table
`news_cache` still exists for backward compatibility, but the active digest logic now relies on `news_digest_pool` and `news_digests`.

## News Digest Rules
Current digest rules are important and should be preserved unless explicitly changed:
- target final digest size: `10`
- candidate target size: `14-16`
- maximum `2` items per domain
- a digest is considered `ready` only when:
  - exactly `10` items exist
  - all items have full original URLs
  - domain spread is acceptable
  - relative-date artifacts are removed
- active pool items are normalized again on read, so stale low-quality records do not bypass newer filters

Rendering format:
- each item is rendered locally, not trusted from model formatting

## Current Integrations
### OpenAI
The project uses:
- `OpenAI Responses API`
- `web_search` tool

Key behavior:
- migration/current-policy questions should use live search when needed
- normal chat and news digest use different prompting strategies
- news collection expects structured JSON, not prose

### Telegram Bot API
The bot sends:
- messages
- keyboards
- `typing` chat actions for long-running flows such as admin digest refresh

### n8n
`n8n` is the current scheduler for digest refresh.

Expected setup:
- `Schedule Trigger`
- `HTTP Request`
- `GET https://<domain>/tasks/refresh-news-digest?lang=ru`
- header: `X-News-Cron-Token`

The scheduler is external because Railway native cron was not used as the primary mechanism in this project state.

## Build, Run, and Verification Commands
Create a virtual environment:
- `python -m venv .venv && source .venv/bin/activate`
- on Windows: `./.venv/Scripts/activate`

Install dependencies:
- `pip install -r requirements.txt`

Initialize or update schema:
- `python init_db.py`

Run locally:
- `python bot_grs.py`

Syntax check before deploy:
- `python3 -m py_compile bot_grs.py init_db.py database.py`

## Coding Style & Editing Rules
- Use Python 3.11+ compatible syntax
- Follow PEP 8 and descriptive snake_case
- Keep logging useful and non-sensitive
- Prefer small helpers for external integrations and persistence logic
- When changing digest behavior, keep model prompt and local filtering in sync
- Do not trust model formatting for final user-facing digest output; render locally

## Testing Guidance
Automated tests are still expected under `tests/` when added, ideally with:
- OpenAI response stubs
- Telegram webhook payload fixtures
- Postgres fixtures / mocks
- regressions for digest normalization and duplicate filtering

Manual verification remains mandatory after changes:
- ordinary text question flow
- Telegram menu buttons
- `/refresh_news` admin flow
- `n8n` cron endpoint
- digest rendering in Telegram

## Commit & PR Notes
Prefer short present-tense commits, but include scope when helpful, for example:
- `news: normalize digest items from active pool`
- `db: add news digest pool table`
- `bot: route button news through active digest`

When changing infra or schema, mention:
- new env vars
- new tables / columns / indexes
- any required n8n or Railway configuration changes

## Security & Operations
- Never hardcode API tokens
- Keep secrets in `.env` locally and Railway variables remotely
- Do not log raw sensitive user data unnecessarily
- When debugging digest refresh, prefer status counts and item/domain totals over dumping full user content
- If DBeaver or manual SQL is used, prefer additive schema fixes over destructive resets
